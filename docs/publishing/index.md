# Publishing Open Source Projects

Publishing code as open source is not the end of a process — it's the beginning. When we release a project publicly, we're committing to transparency, community stewardship, and the kind of ongoing collaboration that builds resilient government services. Code that one agency uses to serve its community can become a foundation for others facing similar challenges.

This section walks through the full lifecycle of taking Nava code from internal development to a public open source project.

## The Publishing Process

1. **[Prepare](prepare.md)**: Review code for public release, set up required files, audit dependencies
2. **[Get Approval](approval.md)**: Internal review, stakeholder sign-off, security review
3. **[Publish](publish.md)**: Create the repository, set up community infrastructure, announce
4. **[Manage Releases](release.md)**: Versioning, changelogs, and ongoing maintenance

## Working in the Open

Nava believes in an "open by default" approach to government technology. This means:

- New projects should be started as public repositories whenever possible
- Code written for government agencies is often intended to be public
- Transparency builds trust with the agencies and communities we serve

Not everything can or should be open source. Code that contains sensitive configuration, proprietary integrations, or information that could compromise security should remain private. When in doubt, consult with your engineering lead.

## Government Context

Open sourcing government technology has unique considerations:

- **Public benefit alignment**: Our work exists to serve people, and open source is a natural extension of that mission
- **Reusability**: Code that helps one agency deliver benefits can help others facing similar challenges
- **Accountability**: Public code can be reviewed by the communities it serves
- **Contract awareness**: Work with your agency partner and Nava's contracts team to understand IP terms before publishing
