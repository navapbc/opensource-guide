# Supporting Your Community

Once people start using your project, they'll have questions, find bugs, and request features. How you respond to your community shapes not just the project's health, but the experience of everyone who depends on it — and Nava's reputation as a thoughtful partner in the open source ecosystem.

## Responding to Issues and PRs

### Response Time
- Acknowledge new issues and PRs promptly, even if you can't address them immediately
- A simple "Thanks for reporting this — we'll take a look" goes a long way
- Set expectations in your CONTRIBUTING.md about typical response times

### Quality of Response
- Be specific and helpful — don't just close issues without explanation
- If you can't reproduce a bug, ask clear follow-up questions
- If a feature request doesn't fit the project's direction, explain why respectfully

### Prioritization
- Security issues should be addressed urgently
- Bug fixes generally take priority over new features
- Label issues to help contributors find things they can work on

## Onboarding New Contributors

Make it easy for new people to contribute:

- Maintain a set of `good first issue` labels for approachable tasks
- Write clear setup instructions that work on common development environments
- Respond to first-time contributors with extra care and encouragement
- Consider creating a "Getting Started" guide for contributors beyond the CONTRIBUTING.md

## Setting Expectations

Be honest about the project's status and your capacity:

- If you can only review PRs weekly, say so
- If the project is in maintenance mode, communicate that clearly
- If certain types of contributions aren't being accepted, explain why
- Update the README if the project's status changes

Consistency matters more than speed. A project that responds slowly but reliably builds more trust than one that's responsive sometimes and silent other times.
