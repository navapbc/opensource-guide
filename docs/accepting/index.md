# Accepting Contributions

When you publish an open source project, you're inviting others to help build something that serves people. Accepting contributions well — setting clear expectations, reviewing thoughtfully, and welcoming new contributors — is what turns a published repository into a community that makes the work better for everyone it reaches.

## Setting Up for Contributions

### Essential Files
Every Nava open source project should have:

- **CONTRIBUTING.md**: How to contribute, what's expected, and how the review process works
- **CODE_OF_CONDUCT.md**: Behavioral expectations for all participants
- **Issue templates**: Structured bug reports and feature requests in `.github/ISSUE_TEMPLATE/`
- **Good first issues**: Label issues that are suitable for newcomers

### Communication
- Be clear about response time expectations (e.g., "We aim to review PRs within one week")
- State what kinds of contributions you're looking for
- If the project isn't actively seeking contributions, say so honestly

## Reviewing External Contributions

### Be Welcoming
- Thank contributors for their time and effort
- Assume good intent, especially from first-time contributors
- Provide constructive, specific feedback — not just "this needs changes"

### Code Review Standards
Apply the same quality standards as internal code:

- **Functionality**: Does it work as described?
- **Security**: Does it introduce any vulnerabilities?
- **Testing**: Are there adequate tests?
- **Documentation**: Is the change documented where needed?
- **Style**: Does it follow the project's conventions?

### Handling Contributions You Can't Accept
Sometimes a well-intentioned contribution doesn't fit. When declining:

- Explain the reason clearly and specifically
- Suggest alternative approaches if possible
- Thank the contributor for their effort
- Close the PR or issue respectfully

## Contributor License Agreements

Decide whether your project requires a CLA:

- **When to require one**: If your project may be relicensed or if your government contract requires clear IP ownership
- **When to skip one**: For most community projects, the license terms of the project (e.g., Apache 2.0) are sufficient — the act of submitting a PR under the project's license implies agreement
- **Developer Certificate of Origin (DCO)**: A lighter-weight alternative where contributors sign off that they have the right to submit their contribution

## Managing External Collaborators

- Use GitHub's permission model thoughtfully — don't grant write access broadly
- Triage permissions allow external contributors to manage issues without write access
- Consider adding active, trusted contributors as collaborators over time

## Measuring Community Health

Track these signals to understand your project's community:

- **Time to first response** on new issues and PRs
- **PR merge rate**: What percentage of external PRs get merged?
- **Contributor diversity**: How many unique contributors are there?
- **Issue resolution time**: How quickly are bugs fixed?
- **Returning contributors**: Do people come back to contribute again?
