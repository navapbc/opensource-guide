# Contributing to Open Source Projects

The tools and frameworks we use to build government services are maintained by communities of people around the world. Contributing back to those projects strengthens the foundations we all build on, deepens our relationships across the civic tech ecosystem, and reflects Nava's commitment to working in the open.

This section covers how Nava team members can contribute to projects maintained by others — and why it matters.

## Why Contribute

- **Improve the tools people depend on**: When we fix bugs or add features upstream, every team and agency using those tools benefits — including the people they serve
- **Good citizenship**: Contributing back to projects we use builds the kind of ecosystem we want to be part of
- **Knowledge sharing**: External contributions expose us to different approaches and broaden our skills
- **Community presence**: Active contribution builds trust and relationships across the civic tech and open source communities

## Before You Contribute

### Check the Project's Guidelines
- Read the project's `CONTRIBUTING.md` file
- Understand their code review process, coding standards, and communication norms
- Check if they use specific tools or workflows (e.g., commit message formats, CI requirements)

### Assess the Project
- Is the project actively maintained?
- Are external contributions welcomed and reviewed in a reasonable time?
- Is there a code of conduct?

### Review Your Contribution
Before submitting:
- Ensure your contribution contains no proprietary Nava code, internal references, or secrets
- Verify the contribution is your own work or properly attributed
- Check that the code quality meets the project's standards

## Contributor License Agreements

Some projects require a Contributor License Agreement (CLA) before accepting contributions. A CLA typically:

- Confirms you have the right to submit the contribution
- Grants the project a license to use your contribution
- May assign copyright or grant a patent license

**Before signing a CLA**, check with your engineering lead. Most CLAs for individual contributions are straightforward, but some corporate CLAs may need review.

## Representing Nava

When contributing to external projects:

- **Be professional**: Your interactions reflect on Nava. Be constructive, patient, and respectful
- **Don't over-represent**: Contribute as an individual unless specifically acting on Nava's behalf
- **Use your judgment**: If a contribution relates to Nava's government work, ensure there are no contract restrictions on sharing the code
- **Follow the project's norms**: Adapt to each project's communication style and processes

## Types of Contributions

Contributions aren't limited to code. Consider:

- **Bug reports**: Well-documented bug reports are valuable
- **Documentation**: Improve READMEs, tutorials, or API docs
- **Code reviews**: Thoughtful reviews help project maintainers
- **Issue triage**: Help categorize and reproduce reported issues
- **Translations**: Help projects reach broader audiences
